1. Introduction: The Plugin System

Jcurl ships with a lightweight plugin system that lets you extend the application without modifying its source. A plugin is a single .java source file annotated with @JcurlPlugin that implements one or more extension point interfaces. Plugins are compiled at runtime by an in-memory Java compiler (JavaSourceCompiler) — no pre-compilation, build steps, or JAR packaging are required.

All extension points live in the package com.jcurl.plugin.extension and extend the marker interface com.jcurl.plugin.ExtensionPoint. When a plugin is loaded, the PluginManager discovers every implemented extension point and registers it in the ExtensionRegistry, where the host application calls them at the appropriate moment.

There are four extension points:

Extension PointInterfaceWhen it runs
Request InterceptorRequestInterceptorBefore an HTTP request is sent
Response InterceptorResponseInterceptorAfter an HTTP response is received
Variable FunctionVariableFunctionExtensionWhen a {{$function}} variable is resolved
Metrics CollectorMetricsCollectorExtensionAfter each sample in a performance test
A single plugin class may implement any combination of these interfaces — for example, one class can intercept requests, register custom variables, and collect metrics all at once.

Package overview of the plugin API:

com.jcurl.plugin
  ├── JcurlPlugin            (annotation)
  ├── ExtensionPoint         (marker interface)
  ├── PluginContext          (host services)
  └── extension
       ├── RequestInterceptor
       ├── ResponseInterceptor
       ├── VariableFunctionExtension
       └── MetricsCollectorExtension

com.jcurl.model
  ├── dto
  │    ├── RequestConfig
  │    ├── ResponseData
  │    └── TimingMetrics
  └── component
       ├── Header
       ├── QueryParam
       ├── RequestBody
       └── AuthConfig

2. The @JcurlPlugin Annotation

Every plugin's main class must be annotated with @JcurlPlugin (package com.jcurl.plugin). The annotation provides metadata shown in the Plugin Manager UI and is retained at runtime (@Retention(RUNTIME), @Target(TYPE)).

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface JcurlPlugin
AttributeTypeDefaultDescription
nameString""Human-readable plugin name
descriptionString""Short description shown in the Plugin Manager
versionString"1.0.0"Semantic version
authorString""Author name
enabledbooleantrueWhether the plugin is enabled on load (can be toggled at runtime)

Minimal example

import com.jcurl.plugin.model.component.Header;
import com.jcurl.plugin.model.dto.RequestConfig;
import com.jcurl.plugin.JcurlPlugin;
import com.jcurl.plugin.PluginContext;
import com.jcurl.plugin.extension.RequestInterceptor;

@JcurlPlugin(
        name = "Sign Plugin",
        description = "Automatically adds an API signature header",
        version = "1.0.0",
        author = "Jcurl"
)
public class SignPlugin implements RequestInterceptor {
    @Override
    public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
        config.getHeaders().add(new Header("X-Sign", "abc123"));
        return config;
    }
}

3. PluginContext Interface

The PluginContext interface (package com.jcurl.plugin) is the plugin's window into the host application. It is the minimal shared contract implemented by all versions (core/JavaFX/Swing), so code written against it works everywhere. The core version additionally exposes CorePluginContext, which extends PluginContext with extra services.

public interface PluginContext
MethodReturnsDescription
getDataDir()PathData storage root directory (defaults to ./.api-client)
getPluginDataDir(String pluginId)PathPer-plugin private data directory (isolated from other plugins)
log(String level, String message)voidWrite a log entry. Plugins should use this instead of SLF4J directly.
log(String level, String message, Throwable t)voidLog a message together with an exception

Recognized level values include "info", "debug", "warn", and "error". Example usage:

@Override
public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
    Path dataDir = ctx.getDataDir();                 // ./.api-client
    Path myDir   = ctx.getPluginDataDir("SignPlugin"); // ./.api-client/plugins/data/SignPlugin
    ctx.log("info", "Sending request to " + config.getUrl());
    try {
        // ... do work ...
    } catch (Exception e) {
        ctx.log("error", "Sign plugin failed", e);
    }
    return config;
}
Use getPluginDataDir() to persist per-plugin state (caches, tokens, config files). Each plugin gets its own isolated directory, so you never collide with another plugin's data.

4. RequestInterceptor & RequestConfig

The RequestInterceptor extension point intercepts an HTTP request before it is sent. Typical use cases: adding a signature/auth header, request logging, encrypting or transforming the body, injecting a tracing ID.

Interceptors run as a chain of responsibility in plugin-load order: the RequestConfig returned by one interceptor is passed to the next.

public interface RequestInterceptor extends ExtensionPoint
    RequestConfig beforeRequest(RequestConfig config, PluginContext ctx)

beforeRequest

ParameterTypeDescription
configRequestConfigThe final, variable-resolved, collection-merged request — mutable
ctxPluginContextHost services
returnsRequestConfigThe (usually same, mutated) config to send

RequestConfig model

Class com.jcurl.plugin.model.dto.RequestConfig is the transient, ready-to-execute request configuration (distinct from the persisted RequestNode). Its mutable fields are accessed via getters/setters:

FieldTypeGetter
methodStringgetMethod() (default "GET")
urlStringgetUrl()
paramsList<QueryParam>getParams()
headersList<Header>getHeaders()
bodyRequestBodygetBody()
authAuthConfiggetAuth()

Header model

Class com.jcurl.plugin.model.component.Header — a key/value pair with an enabled flag and optional description.

new Header(String key, String value)                 // enabled = true
new Header(String key, String value, boolean enabled)
// getters: getKey(), getValue(), isEnabled(), getDescription()

Example: add a tracing header unless the user already set one

@Override
public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
    String requestId = java.util.UUID.randomUUID().toString().substring(0, 8);

    boolean hasId = config.getHeaders().stream()
            .anyMatch(h -> "X-Request-Id".equalsIgnoreCase(h.getKey()));
    if (!hasId) {
        config.getHeaders().add(new Header("X-Request-Id", requestId));
    }
    config.getHeaders().add(new Header("X-Plugin-Timestamp",
            String.valueOf(System.currentTimeMillis())));

    ctx.log("info", String.format("Sending %s %s (RequestId=%s)",
            config.getMethod(), config.getUrl(), requestId));
    return config;
}

5. ResponseInterceptor & ResponseData

The ResponseInterceptor extension point intercepts an HTTP response after it is received. Typical use cases: response decryption, response logging, automatically extracting a token, transforming the body, marking error responses, running assertions.

Like request interceptors, response interceptors form a chain in plugin-load order.

public interface ResponseInterceptor extends ExtensionPoint
    ResponseData afterResponse(ResponseData response, RequestConfig config, PluginContext ctx)

afterResponse

ParameterTypeDescription
responseResponseDataThe received response — mutable
configRequestConfigThe original request config — read-only
ctxPluginContextHost services
returnsResponseDataThe (usually same, mutated) response

ResponseData model

Class com.jcurl.plugin.model.dto.ResponseData holds the complete response. Key accessors:

FieldTypeGetter
statusCodeintgetStatusCode() (e.g. 200)
statusTextStringgetStatusText() (e.g. "OK")
protocolStringgetProtocol() (e.g. "HTTP/1.1")
headersList<Header>getHeaders()
bodyStringgetBody() (text body; binary uses binaryBase64)
contentTypeStringgetContentType()
sizelonggetSize() (bytes)
timingTimingMetricsgetTiming() (e.g. getTotalMs())
cancelledbooleanisCancelled()
errorStringgetError() (failure message, null on success)
binaryBase64StringgetBinaryBase64() (Base64 for image/binary previews)
—booleanisSuccess() (true if no error, not cancelled, status > 0)

Example: log timing and mark errors

@Override
public ResponseData afterResponse(ResponseData response, RequestConfig config, PluginContext ctx) {
    String timing = response.getTiming() != null
            ? response.getTiming().getTotalMs() + "ms"
            : "unknown";
    ctx.log("info", String.format("Got %d %s (%s, %d bytes)",
            response.getStatusCode(), response.getStatusText(),
            timing, response.getSize()));

    // Detect a token in a JSON body (you could extract it to an env variable)
    if (response.getBody() != null && response.getBody().contains("\"token\"")) {
        ctx.log("debug", "Response contains a token field");
    }

    // Tag error responses
    if (response.getStatusCode() >= 400) {
        response.getHeaders().add(new Header("X-Plugin-Error-Marked", "true"));
    }
    return response;
}

6. VariableFunctionExtension

This extension point registers custom dynamic variables of the form {{$functionName}} that can be used in URLs, headers, and request bodies — exactly like the built-in functions {{$timestamp}}, {{$uuid}}, {{$randomInt}}, and {{$datetime}}.

public interface VariableFunctionExtension extends ExtensionPoint
    List<String> getFunctionNames(PluginContext ctx)
    String executeFunction(String functionName, String args, PluginContext ctx)

getFunctionNames

Returns the function names without the {{$}} wrapping. Returning ["md5", "base64"] makes {{$md5}} and {{$base64}} available (and they appear in the variable auto-completion popup).

executeFunction

ParameterTypeDescription
functionNameStringThe function name (without {{$}})
argsStringText inside the parentheses, already stripped of the parens; empty string when no argument
ctxPluginContextHost services
returnsStringThe computed value, or null if the function is not handled

For example, {{$md5(Hello)}} calls executeFunction("md5", "Hello", ctx). {{$uuid2}} calls it with "" as args.

Example: md5, base64, uuid2

@Override
public List<String> getFunctionNames(PluginContext ctx) {
    return List.of("md5", "base64", "uuid2");
}

@Override
public String executeFunction(String functionName, String args, PluginContext ctx) {
    try {
        switch (functionName) {
            case "md5":
                return md5(args != null ? args : "");
            case "base64":
                return Base64.getEncoder().encodeToString(
                        (args != null ? args : "").getBytes(StandardCharsets.UTF_8));
            case "uuid2":
                return UUID.randomUUID().toString().replace("-", "");
            default:
                return null;   // function not supported by this plugin
        }
    } catch (Exception e) {
        ctx.log("error", "Function " + functionName + " failed: " + e.getMessage());
        return null;
    }
}
Returning null from executeFunction signals "not handled", so multiple variable-function plugins can coexist and each only answers its own names.

7. MetricsCollectorExtension

This extension point collects custom metrics during a performance/load test. Jcurl already records built-in metrics (DNS/TCP/TLS/TTFB/Total timing, status code, response size); your plugin can add business-level metrics such as memory usage, CPU load, or a derived success flag. Custom metric names appear in the performance test report alongside the built-in ones.

public interface MetricsCollectorExtension extends ExtensionPoint
    List<String> getMetricNames()
    Map<String, Double> collectMetrics(RequestConfig config, ResponseData response, PluginContext ctx)

getMetricNames

Returns the list of metric names this collector provides (e.g. ["response_kb", "is_success"]). These names show up as columns in the performance test report.

collectMetrics

Called once per request after it completes during a performance test. Returns a map of metric name to numeric value.

Example: response size in KB and a success flag

@Override
public List<String> getMetricNames() {
    return List.of("response_kb", "is_success");
}

@Override
public Map<String, Double> collectMetrics(RequestConfig config, ResponseData response, PluginContext ctx) {
    Map<String, Double> metrics = new HashMap<>();

    double kb = response.getSize() / 1024.0;
    metrics.put("response_kb", Math.round(kb * 100.0) / 100.0);

    int code = response.getStatusCode();
    metrics.put("is_success", (code >= 200 && code < 400) ? 1.0 : 0.0);

    return metrics;
}

8. Complete Working Example (DemoPlugin.java)

The following plugin implements all four extension points at once. Drop it into ./.api-client/plugins/ and it will be compiled and loaded on the next plugin scan.

import com.jcurl.plugin.model.component.Header;
import com.jcurl.plugin.model.dto.RequestConfig;
import com.jcurl.plugin.model.dto.ResponseData;
import com.jcurl.plugin.ExtensionPoint;
import com.jcurl.plugin.JcurlPlugin;
import com.jcurl.plugin.PluginContext;
import com.jcurl.plugin.extension.MetricsCollectorExtension;
import com.jcurl.plugin.extension.RequestInterceptor;
import com.jcurl.plugin.extension.ResponseInterceptor;
import com.jcurl.plugin.extension.VariableFunctionExtension;

import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.util.Base64;
import java.util.HashMap;
import java.util.List;
import java.util.Map;
import java.util.UUID;

/**
 * Jcurl demo plugin - demonstrates the four extension points.
 *
 * Features:
 *   - RequestInterceptor: adds X-Request-Id and X-Timestamp headers, logs requests
 *   - ResponseInterceptor: logs status/timing, detects a token in the body, marks errors
 *   - VariableFunctionExtension: adds {{$md5}}, {{$base64}}, {{$uuid2}}
 *   - MetricsCollectorExtension: collects response_kb and is_success metrics
 *
 * Install: copy this file to ./.api-client/plugins/ or use the Plugin Manager.
 */
@JcurlPlugin(
        name = "Demo Plugin",
        description = "Demonstrates request/response interceptors, variable functions, and metrics",
        version = "1.0.0",
        author = "Jcurl"
)
public class DemoPlugin implements RequestInterceptor, ResponseInterceptor,
        VariableFunctionExtension, MetricsCollectorExtension {

    private static final String HEADER_REQUEST_ID = "X-Request-Id";
    private static final String HEADER_TIMESTAMP = "X-Plugin-Timestamp";

    // ==================== RequestInterceptor ====================

    @Override
    public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
        String requestId = UUID.randomUUID().toString().substring(0, 8);

        boolean hasRequestId = config.getHeaders().stream()
                .anyMatch(h -> HEADER_REQUEST_ID.equalsIgnoreCase(h.getKey()));
        if (!hasRequestId) {
            config.getHeaders().add(new Header(HEADER_REQUEST_ID, requestId));
        }

        config.getHeaders().add(new Header(HEADER_TIMESTAMP,
                String.valueOf(System.currentTimeMillis())));

        ctx.log("info", String.format("[Demo] Sending %s %s (RequestId=%s)",
                config.getMethod(), config.getUrl(), requestId));
        return config;
    }

    // ==================== ResponseInterceptor ====================

    @Override
    public ResponseData afterResponse(ResponseData response, RequestConfig config, PluginContext ctx) {
        String timingInfo = response.getTiming() != null
                ? response.getTiming().getTotalMs() + "ms"
                : "unknown";
        ctx.log("info", String.format("[Demo] Got %d %s (%s, %d bytes)",
                response.getStatusCode(), response.getStatusText(),
                timingInfo, response.getSize()));

        if (response.getBody() != null && response.getBody().contains("\"token\"")) {
            ctx.log("debug", "[Demo] Response body contains a token field");
        }

        if (response.getStatusCode() >= 400) {
            response.getHeaders().add(new Header("X-Plugin-Error-Marked", "true"));
        }
        return response;
    }

    // ==================== VariableFunctionExtension ====================

    @Override
    public List<String> getFunctionNames(PluginContext ctx) {
        return List.of("md5", "base64", "uuid2");
    }

    @Override
    public String executeFunction(String functionName, String args, PluginContext ctx) {
        try {
            switch (functionName) {
                case "md5":
                    return md5(args != null ? args : "");
                case "base64":
                    return Base64.getEncoder().encodeToString(
                            (args != null ? args : "").getBytes(StandardCharsets.UTF_8));
                case "uuid2":
                    return UUID.randomUUID().toString().replace("-", "");
                default:
                    return null;
            }
        } catch (Exception e) {
            ctx.log("error", "[Demo] Function " + functionName + " failed: " + e.getMessage());
            return null;
        }
    }

    private String md5(String input) throws Exception {
        MessageDigest md = MessageDigest.getInstance("MD5");
        byte[] digest = md.digest(input.getBytes(StandardCharsets.UTF_8));
        StringBuilder sb = new StringBuilder();
        for (byte b : digest) {
            sb.append(String.format("%02x", b));
        }
        return sb.toString();
    }

    // ==================== MetricsCollectorExtension ====================

    @Override
    public List<String> getMetricNames() {
        return List.of("response_kb", "is_success");
    }

    @Override
    public Map<String, Double> collectMetrics(RequestConfig config, ResponseData response, PluginContext ctx) {
        Map<String, Double> metrics = new HashMap<>();

        double kb = response.getSize() / 1024.0;
        metrics.put("response_kb", Math.round(kb * 100.0) / 100.0);

        int code = response.getStatusCode();
        metrics.put("is_success", (code >= 200 && code < 400) ? 1.0 : 0.0);

        return metrics;
    }
}

9. Installing Plugins

Plugins are plain .java source files. Jcurl compiles them in memory at load time using the JDK compiler, so there is no build step. The data directory defaults to ./.api-client and plugins live in ./.api-client/plugins/.

Option A — Copy the file manually

# Copy your plugin source into the plugins directory
cp DemoPlugin.java ./.api-client/plugins/

# Reload plugins from the UI (Plugin Manager -> Reload),
# or simply restart Jcurl. The plugin is compiled on first load.

Option B — Use the Plugin Manager

  1. Open the Plugin Manager dialog from the GUI.
  2. Click Install and paste your plugin source (or point it at a .java file).
  3. The manager compiles, loads, and enables the plugin immediately. You can also enable/disable, uninstall, and reload plugins from here.

Directory layout

.api-client/
├── settings.json
├── plugins/
│   ├── DemoPlugin.java          <- your plugin source files
│   └── data/
│       └── Demo Plugin/         <- per-plugin private data (via getPluginDataDir)
└── ...
A JDK (not just a JRE) is required because plugins are compiled at runtime. When launching the JavaFX GUI you must add --add-modules jdk.compiler so the compiler module is available.
Compilation errors are reported in the Plugin Manager and the application log. Fix the source and click Reload — no restart is needed for source-only plugins.

10. Cross-Version Compatibility

The plugin API is designed to be identical across all three editions of Jcurl:

EditionModuleContext implementation
CLI (core)jcurl-coreCorePluginContext (extends PluginContext)
JavaFX GUIjcurl-javafxImplements PluginContext
Swing GUI (Win7)jcurl-swingSwingPluginContext (implements PluginContext)

What this means in practice:

  • The same .java plugin file works unchanged in the CLI, the JavaFX GUI, and the Swing GUI.
  • The four extension points, the @JcurlPlugin annotation, the PluginContext methods, and the model classes (RequestConfig, ResponseData, Header) are the same contract everywhere.
  • If you only depend on the base PluginContext interface, your plugin is portable to every edition. Avoid casting to CorePluginContext unless you only target the core/CLI edition and need its extra services.
  • Plugins that only touch request/response data and log through ctx.log() are fully portable.
Write plugins against the PluginContext interface, not a specific implementation. That single rule keeps your plugin working across Swing, JavaFX, and CLI.

Ready to write your first plugin? Copy the DemoPlugin above into ./.api-client/plugins/, reload, and watch your interceptors, variables, and metrics come to life.

1. 插件系统简介

Jcurl 内置一套轻量级插件系统，无需修改源码即可扩展应用功能。一个插件就是一个用 @JcurlPlugin 标注、并实现一个或多个扩展点接口的 .java 源文件。插件由内置内存编译器 (JavaSourceCompiler) 在运行时编译——无需预编译、构建步骤或打包 JAR。

所有扩展点都位于包 com.jcurl.plugin.extension，并继承标记接口 com.jcurl.plugin.ExtensionPoint。插件加载时，PluginManager 会发现它实现的所有扩展点并注册到 ExtensionRegistry，宿主程序在合适时机调用它们。

共有 四大扩展点：

扩展点接口触发时机
请求拦截器RequestInterceptorHTTP 请求发送前
响应拦截器ResponseInterceptorHTTP 响应接收后
变量函数VariableFunctionExtension解析 {{$function}} 变量时
指标采集MetricsCollectorExtension性能测试每次采样完成后
单个插件类可以实现这些接口的任意组合——例如一个类同时拦截请求、注册自定义变量并采集指标。

插件 API 的包结构概览：

3. PluginContext 接口

PluginContext 接口（包 com.jcurl.plugin）是插件访问宿主程序的窗口。它是所有版本（core/JavaFX/Swing）共同实现的最小共享契约，因此基于它编写的代码到处可用。core 版本额外提供了 CorePluginContext，它扩展 PluginContext 暴露更多服务。

public interface PluginContext
方法返回类型说明
getDataDir()Path数据存储根目录（默认 ./.api-client）
getPluginDataDir(String pluginId)Path插件专属数据目录（与其他插件隔离）
log(String level, String message)void记录日志。插件应使用此方法，而非直接用 SLF4J。
log(String level, String message, Throwable t)void记录日志并附带异常

支持的 level 取值包括 "info"、"debug"、"warn"、"error"。用法示例：

@Override
public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
    Path dataDir = ctx.getDataDir();                  // ./.api-client
    Path myDir   = ctx.getPluginDataDir("SignPlugin"); // ./.api-client/plugins/data/SignPlugin
    ctx.log("info", "发送请求到 " + config.getUrl());
    try {
        // ... 执行工作 ...
    } catch (Exception e) {
        ctx.log("error", "签名插件失败", e);
    }
    return config;
}
使用 getPluginDataDir() 持久化插件私有状态（缓存、令牌、配置文件）。每个插件都有独立隔离的目录，不会与其他插件数据冲突。

4. RequestInterceptor 与 RequestConfig

RequestInterceptor 扩展点在 HTTP 请求发送前拦截。典型场景：添加签名/认证头、请求日志、加密或转换请求体、注入链路追踪 ID。

拦截器按插件加载顺序以责任链方式执行：前一个拦截器返回的 RequestConfig 会传给下一个。

public interface RequestInterceptor extends ExtensionPoint
    RequestConfig beforeRequest(RequestConfig config, PluginContext ctx)

beforeRequest

参数类型说明
configRequestConfig变量已解析、集合配置已合并的最终请求——可修改
ctxPluginContext宿主服务
返回RequestConfig要发送的（通常是同一个被修改过的）配置

RequestConfig 模型

类 com.jcurl.plugin.model.dto.RequestConfig 是执行时的瞬时请求配置（区别于持久化的 RequestNode）。其可变字段通过 getter/setter 访问：

字段类型Getter
methodStringgetMethod()（默认 "GET"）
urlStringgetUrl()
paramsList<QueryParam>getParams()
headersList<Header>getHeaders()
bodyRequestBodygetBody()
authAuthConfiggetAuth()

Header 模型

类 com.jcurl.plugin.model.component.Header——带 enabled 标志和可选 description 的键值对。

new Header(String key, String value)                 // enabled = true
new Header(String key, String value, boolean enabled)
// getter: getKey(), getValue(), isEnabled(), getDescription()

示例：若用户未设置则添加追踪头

@Override
public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
    String requestId = java.util.UUID.randomUUID().toString().substring(0, 8);

    boolean hasId = config.getHeaders().stream()
            .anyMatch(h -> "X-Request-Id".equalsIgnoreCase(h.getKey()));
    if (!hasId) {
        config.getHeaders().add(new Header("X-Request-Id", requestId));
    }
    config.getHeaders().add(new Header("X-Plugin-Timestamp",
            String.valueOf(System.currentTimeMillis())));

    ctx.log("info", String.format("发送 %s %s (RequestId=%s)",
            config.getMethod(), config.getUrl(), requestId));
    return config;
}

5. ResponseInterceptor 与 ResponseData

ResponseInterceptor 扩展点在 HTTP 响应接收后拦截。典型场景：响应解密、响应日志、自动提取 token、转换响应体、标记错误响应、断言校验。

与请求拦截器一样，响应拦截器按插件加载顺序组成责任链。

public interface ResponseInterceptor extends ExtensionPoint
    ResponseData afterResponse(ResponseData response, RequestConfig config, PluginContext ctx)

afterResponse

参数类型说明
responseResponseData接收到的响应——可修改
configRequestConfig原始请求配置——只读
ctxPluginContext宿主服务
返回ResponseData（通常是同一个被修改过的）响应

ResponseData 模型

类 com.jcurl.plugin.model.dto.ResponseData 持有完整响应。主要访问器：

字段类型Getter
statusCodeintgetStatusCode()（如 200）
statusTextStringgetStatusText()（如 "OK"）
protocolStringgetProtocol()（如 "HTTP/1.1"）
headersList<Header>getHeaders()
bodyStringgetBody()（文本响应体；二进制用 binaryBase64）
contentTypeStringgetContentType()
sizelonggetSize()（字节）
timingTimingMetricsgetTiming()（如 getTotalMs()）
cancelledbooleanisCancelled()
errorStringgetError()（失败信息，成功时为 null）
binaryBase64StringgetBinaryBase64()（图片/二进制预览的 Base64）
—booleanisSuccess()（无错误、未取消、状态码 > 0 时为 true）

示例：记录耗时并标记错误

@Override
public ResponseData afterResponse(ResponseData response, RequestConfig config, PluginContext ctx) {
    String timing = response.getTiming() != null
            ? response.getTiming().getTotalMs() + "ms"
            : "未知";
    ctx.log("info", String.format("收到 %d %s (%s, %d 字节)",
            response.getStatusCode(), response.getStatusText(),
            timing, response.getSize()));

    // 检测 JSON 响应体中的 token（可提取到环境变量）
    if (response.getBody() != null && response.getBody().contains("\"token\"")) {
        ctx.log("debug", "响应中包含 token 字段");
    }

    // 标记错误响应
    if (response.getStatusCode() >= 400) {
        response.getHeaders().add(new Header("X-Plugin-Error-Marked", "true"));
    }
    return response;
}

6. VariableFunctionExtension

此扩展点注册 {{$functionName}} 形式的自定义动态变量，可在 URL、请求头、请求体中使用——与内置函数 {{$timestamp}}、{{$uuid}}、{{$randomInt}}、{{$datetime}} 完全一致。

public interface VariableFunctionExtension extends ExtensionPoint
    List<String> getFunctionNames(PluginContext ctx)
    String executeFunction(String functionName, String args, PluginContext ctx)

getFunctionNames

返回不带 {{$}} 包裹的函数名列表。返回 ["md5", "base64"] 即可使用 {{$md5}} 和 {{$base64}}（并出现在变量自动补全弹窗中）。

executeFunction

参数类型说明
functionNameString函数名（不带 {{$}}）
argsString括号内的内容，已去除括号；无参数时为空字符串
ctxPluginContext宿主服务
返回String计算结果；不处理该函数时返回 null

例如 {{$md5(Hello)}} 会调用 executeFunction("md5", "Hello", ctx)；{{$uuid2}} 则以空字符串 "" 作为 args 调用。

示例：md5、base64、uuid2

@Override
public List<String> getFunctionNames(PluginContext ctx) {
    return List.of("md5", "base64", "uuid2");
}

@Override
public String executeFunction(String functionName, String args, PluginContext ctx) {
    try {
        switch (functionName) {
            case "md5":
                return md5(args != null ? args : "");
            case "base64":
                return Base64.getEncoder().encodeToString(
                        (args != null ? args : "").getBytes(StandardCharsets.UTF_8));
            case "uuid2":
                return UUID.randomUUID().toString().replace("-", "");
            default:
                return null;   // 本插件不处理该函数
        }
    } catch (Exception e) {
        ctx.log("error", "函数 " + functionName + " 执行失败: " + e.getMessage());
        return null;
    }
}
executeFunction 返回 null 表示"不处理"，因此多个变量函数插件可共存，各自只响应自己的函数名。

7. MetricsCollectorExtension

此扩展点在性能/压力测试中采集自定义指标。Jcurl 已内置指标（DNS/TCP/TLS/TTFB/Total 耗时、状态码、响应大小）；插件可添加业务级指标，如内存占用、CPU 负载、派生的成功标志。自定义指标名会与内置指标一起出现在性能测试报告中。

public interface MetricsCollectorExtension extends ExtensionPoint
    List<String> getMetricNames()
    Map<String, Double> collectMetrics(RequestConfig config, ResponseData response, PluginContext ctx)

getMetricNames

返回本采集器提供的指标名列表（如 ["response_kb", "is_success"]）。这些名称将作为性能测试报告中的列。

collectMetrics

性能测试中每次请求完成后调用一次，返回 指标名 → 数值 的映射。

示例：响应大小(KB) 与成功标志

@Override
public List<String> getMetricNames() {
    return List.of("response_kb", "is_success");
}

@Override
public Map<String, Double> collectMetrics(RequestConfig config, ResponseData response, PluginContext ctx) {
    Map<String, Double> metrics = new HashMap<>();

    double kb = response.getSize() / 1024.0;
    metrics.put("response_kb", Math.round(kb * 100.0) / 100.0);

    int code = response.getStatusCode();
    metrics.put("is_success", (code >= 200 && code < 400) ? 1.0 : 0.0);

    return metrics;
}

8. 完整示例 (DemoPlugin.java)

下面的插件同时实现全部四个扩展点。把它放入 ./.api-client/plugins/，下次扫描插件时即会编译加载。

import com.jcurl.plugin.model.component.Header;
import com.jcurl.plugin.model.dto.RequestConfig;
import com.jcurl.plugin.model.dto.ResponseData;
import com.jcurl.plugin.ExtensionPoint;
import com.jcurl.plugin.JcurlPlugin;
import com.jcurl.plugin.PluginContext;
import com.jcurl.plugin.extension.MetricsCollectorExtension;
import com.jcurl.plugin.extension.RequestInterceptor;
import com.jcurl.plugin.extension.ResponseInterceptor;
import com.jcurl.plugin.extension.VariableFunctionExtension;

import java.nio.charset.StandardCharsets;
import java.security.MessageDigest;
import java.util.Base64;
import java.util.HashMap;
import java.util.List;
import java.util.Map;
import java.util.UUID;

/**
 * Jcurl 演示插件 — 展示四大扩展点的用法。
 *
 * 功能:
 *   - 请求拦截器: 添加 X-Request-Id 和 X-Timestamp 头, 记录请求日志
 *   - 响应拦截器: 记录状态和耗时, 检测响应中的 token, 标记错误
 *   - 变量函数: 新增 {{$md5}}、{{$base64}}、{{$uuid2}}
 *   - 指标采集: 采集 response_kb 和 is_success
 *
 * 安装: 将此文件复制到 ./.api-client/plugins/ 或通过插件管理器安装。
 */
@JcurlPlugin(
        name = "演示插件",
        description = "展示请求拦截/响应拦截/变量函数/指标采集四大扩展点",
        version = "1.0.0",
        author = "Jcurl"
)
public class DemoPlugin implements RequestInterceptor, ResponseInterceptor,
        VariableFunctionExtension, MetricsCollectorExtension {

    private static final String HEADER_REQUEST_ID = "X-Request-Id";
    private static final String HEADER_TIMESTAMP = "X-Plugin-Timestamp";

    // ==================== 请求拦截器 ====================

    @Override
    public RequestConfig beforeRequest(RequestConfig config, PluginContext ctx) {
        String requestId = UUID.randomUUID().toString().substring(0, 8);

        boolean hasRequestId = config.getHeaders().stream()
                .anyMatch(h -> HEADER_REQUEST_ID.equalsIgnoreCase(h.getKey()));
        if (!hasRequestId) {
            config.getHeaders().add(new Header(HEADER_REQUEST_ID, requestId));
        }

        config.getHeaders().add(new Header(HEADER_TIMESTAMP,
                String.valueOf(System.currentTimeMillis())));

        ctx.log("info", String.format("[演示插件] 发送请求: %s %s (RequestId=%s)",
                config.getMethod(), config.getUrl(), requestId));
        return config;
    }

    // ==================== 响应拦截器 ====================

    @Override
    public ResponseData afterResponse(ResponseData response, RequestConfig config, PluginContext ctx) {
        String timingInfo = response.getTiming() != null
                ? response.getTiming().getTotalMs() + "ms"
                : "未知";
        ctx.log("info", String.format("[演示插件] 收到响应: %d %s (耗时: %s, 大小: %d 字节)",
                response.getStatusCode(),
                response.getStatusText(),
                timingInfo,
                response.getSize()));

        if (response.getBody() != null && response.getBody().contains("\"token\"")) {
            ctx.log("debug", "[演示插件] 检测到响应中包含 token 字段");
        }

        if (response.getStatusCode() >= 400) {
            response.getHeaders().add(new Header("X-Plugin-Error-Marked", "true"));
        }
        return response;
    }

    // ==================== 变量函数扩展 ====================

    @Override
    public List<String> getFunctionNames(PluginContext ctx) {
        return List.of("md5", "base64", "uuid2");
    }

    @Override
    public String executeFunction(String functionName, String args, PluginContext ctx) {
        try {
            switch (functionName) {
                case "md5":
                    return md5(args != null ? args : "");
                case "base64":
                    return Base64.getEncoder().encodeToString(
                            (args != null ? args : "").getBytes(StandardCharsets.UTF_8));
                case "uuid2":
                    return UUID.randomUUID().toString().replace("-", "");
                default:
                    return null;
            }
        } catch (Exception e) {
            ctx.log("error", "[演示插件] 变量函数执行失败: " + functionName + ", 错误: " + e.getMessage());
            return null;
        }
    }

    private String md5(String input) throws Exception {
        MessageDigest md = MessageDigest.getInstance("MD5");
        byte[] digest = md.digest(input.getBytes(StandardCharsets.UTF_8));
        StringBuilder sb = new StringBuilder();
        for (byte b : digest) {
            sb.append(String.format("%02x", b));
        }
        return sb.toString();
    }

    // ==================== 指标采集扩展 ====================

    @Override
    public List<String> getMetricNames() {
        return List.of("response_kb", "is_success");
    }

    @Override
    public Map<String, Double> collectMetrics(RequestConfig config, ResponseData response, PluginContext ctx) {
        Map<String, Double> metrics = new HashMap<>();

        double kb = response.getSize() / 1024.0;
        metrics.put("response_kb", Math.round(kb * 100.0) / 100.0);

        int code = response.getStatusCode();
        metrics.put("is_success", (code >= 200 && code < 400) ? 1.0 : 0.0);

        return metrics;
    }
}

9. 安装插件

插件就是普通的 .java 源文件。Jcurl 加载时使用 JDK 编译器在内存中编译，因此无需构建步骤。数据目录默认 ./.api-client，插件放在 ./.api-client/plugins/。

方式 A — 手动复制文件

# 将插件源码复制到 plugins 目录
cp DemoPlugin.java ./.api-client/plugins/

# 从界面重载插件（插件管理器 -> 重载），
# 或直接重启 Jcurl。插件在首次加载时编译。

方式 B — 使用插件管理器

  1. 从图形界面打开插件管理器对话框。
  2. 点击安装，粘贴插件源码（或指向一个 .java 文件）。
  3. 管理器会立即编译、加载并启用插件。你也可以在此启用/禁用、卸载、重载插件。

目录结构

.api-client/
├── settings.json
├── plugins/
│   ├── DemoPlugin.java          <- 你的插件源文件
│   └── data/
│       └── 演示插件/            <- 插件私有数据 (通过 getPluginDataDir)
└── ...
运行时编译插件需要 JDK（而非仅 JRE）。启动 JavaFX 图形界面时必须加上 --add-modules jdk.compiler，以提供编译器模块。
编译错误会在插件管理器和应用日志中报告。修正源码后点击重载即可——纯源码插件无需重启。

10. 跨版本兼容性

插件 API 在 Jcurl 的全部三个版本中保持一致：

版本模块上下文实现
CLI (core)jcurl-coreCorePluginContext（继承 PluginContext）
JavaFX 图形界面jcurl-javafx实现 PluginContext
Swing 图形界面 (Win7)jcurl-swingSwingPluginContext（实现 PluginContext）

实际含义：

  • 同一个 .java 插件文件在 CLI、JavaFX 图形界面、Swing 图形界面中无需修改即可使用。
  • 四大扩展点、@JcurlPlugin 注解、PluginContext 方法以及模型类（RequestConfig、ResponseData、Header）在各版本中是同一套契约。
  • 只要仅依赖基础 PluginContext 接口，你的插件就可移植到每个版本。除非只面向 core/CLI 版本且需要其额外服务，否则不要强制转换为 CorePluginContext。
  • 只操作请求/响应数据并通过 ctx.log() 记录日志的插件完全可移植。
面向 PluginContext 接口而非具体实现编写插件。仅此一条规则即可让你的插件在 Swing、JavaFX、CLI 中通用。

准备好写第一个插件了吗？把上面的 DemoPlugin 复制到 ./.api-client/plugins/，重载，即可看到你的拦截器、变量和指标生效。
