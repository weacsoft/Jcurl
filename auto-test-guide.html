1. Overview — What Is This?

This document is a specification for AI assistants (and human developers). It teaches you how to generate auto-test import files — structured JSON or text files that describe HTTP API test cases and can be directly imported into Jcurl for execution.

Use case: If you are an AI helping someone build a project (a REST API, a web service, a microservice), you can generate an import file following this spec. The developer then imports it into Jcurl and immediately has all their API endpoints ready to test — no manual configuration needed.

Workflow

  1. AI reads the project's API definitions (routes, controllers, OpenAPI docs, etc.)
  2. AI generates a .json file in Postman v2.1 format following this spec
  3. Developer opens Jcurl → Import → selects the file
  4. All endpoints appear as a collection, ready to execute

2. Supported Formats

Jcurl supports three import formats. Postman v2.1 is strongly recommended for AI-generated files because it supports the most features.

FormatExtensionFeaturesRecommendation
Postman v2.1.jsonFolders, variables, auth, all body types, query params, headersBest
OpenAPI 3.0.jsonPaths by tag, parameters, request body typeGood
cURL.txtSingle request only, headers, body, methodLimited

3. Format 1: Postman v2.1 JSON Recommended

This is the most complete format. It supports nested folders, collection-level variables, per-request auth, and all body types.

3.1 Top-Level Structure

{
  "info": {
    "name": "My API Tests",
    "schema": "https://schema.getpostman.com/json/collection/v2.1.0/collection.json",
    "_postman_id": "any-unique-string"
  },
  "variable": [ ... ],
  "item": [ ... ]
}
FieldTypeRequiredDescription
info.namestringYesCollection name shown in Jcurl
info.schemastringNoPostman schema URL (for compatibility)
info._postman_idstringNoUnique ID (auto-generated if omitted)
variablearrayNoCollection-level variables (see §6)
itemarrayYesTop-level items (folders or requests)

3.2 Item Types: Folder vs Request

Each item in the item array is either a folder (contains nested items) or a request (contains a request object).

Folder

{
  "name": "User Management",
  "item": [
    { "name": "Login", "request": { ... } },
    { "name": "Get Profile", "request": { ... } }
  ]
}

Request

{
  "name": "Create User",
  "request": {
    "method": "POST",
    "url": { ... },
    "header": [ ... ],
    "body": { ... },
    "auth": { ... }
  }
}
Detection rule: If an item has an item array and no request field, it's a folder. If it has a request field, it's a request.

3.3 URL Structure

The url field can be a simple string or an object with query parameters:

Simple URL

"url": "https://api.example.com/users"

URL with Query Parameters

"url": {
  "raw": "https://api.example.com/users?page=1&limit=20",
  "query": [
    { "key": "page", "value": "1" },
    { "key": "limit", "value": "20", "disabled": false }
  ]
}
FieldTypeDescription
rawstringFull URL (used as the request URL)
queryarrayQuery parameter list
query[].keystringParameter name
query[].valuestringParameter value
query[].disabledbooleanIf true, parameter is disabled (default: false)

3.4 Headers

"header": [
  { "key": "Content-Type", "value": "application/json" },
  { "key": "Authorization", "value": "Bearer {{token}}", "disabled": false },
  { "key": "X-Debug", "value": "true", "disabled": true }
]
FieldTypeDescription
keystringHeader name
valuestringHeader value
disabledbooleanIf true, header is disabled (default: false)

3.5 Body Types

The body object uses a mode field to specify the body type:

raw (JSON / XML / Text)

"body": {
  "mode": "raw",
  "raw": "{\"name\":\"Alice\",\"email\":\"alice@example.com\"}",
  "options": {
    "raw": { "language": "json" }
  }
}

The options.raw.language can be json, xml, html, or text.

form-data (Multipart)

"body": {
  "mode": "formdata",
  "formdata": [
    { "key": "username", "value": "alice", "type": "text" },
    { "key": "avatar", "src": "/path/to/file.jpg", "type": "file" },
    { "key": "disabled_field", "value": "x", "type": "text", "disabled": true }
  ]
}

x-www-form-urlencoded

"body": {
  "mode": "urlencoded",
  "urlencoded": [
    { "key": "grant_type", "value": "password" },
    { "key": "username", "value": "alice" },
    { "key": "password", "value": "secret123" }
  ]
}
ModeJcurl Body TypeField NameDescription
rawrawrawRaw text body (JSON/XML/text)
formdataform-dataformdataMultipart form with file support
urlencodedx-www-form-urlencodedurlencodedURL-encoded form data
filebinaryfile.srcBinary file upload

3.6 Authentication

Basic Auth

"auth": {
  "type": "basic",
  "basic": [
    { "key": "username", "value": "admin" },
    { "key": "password", "value": "secret" }
  ]
}

Bearer Token

"auth": {
  "type": "bearer",
  "bearer": [
    { "key": "token", "value": "{{token}}" }
  ]
}

API Key

"auth": {
  "type": "apikey",
  "apikey": [
    { "key": "key", "value": "X-API-Key" },
    { "key": "value", "value": "abc123" },
    { "key": "in", "value": "header" }
  ]
}

The in field can be header or query.

3.7 Collection Variables

Variables defined at the collection level can be referenced anywhere using {{variable_name}} syntax:

"variable": [
  { "key": "base_url", "value": "https://api.example.com" },
  { "key": "token", "value": "" },
  { "key": "user_id", "value": "" }
]

Then in requests:

"url": { "raw": "{{base_url}}/users/{{user_id}}" }

4. Format 2: OpenAPI 3.0 JSON

If your project already has an OpenAPI/Swagger specification, Jcurl can import it directly. Endpoints are grouped by their first tag.

Structure

{
  "openapi": "3.0.0",
  "info": { "title": "My API", "version": "1.0.0" },
  "servers": [
    { "url": "https://api.example.com" }
  ],
  "paths": {
    "/users": {
      "get": {
        "tags": ["Users"],
        "summary": "List all users",
        "parameters": [
          { "name": "page", "in": "query", "description": "Page number" },
          { "name": "X-Trace-Id", "in": "header", "description": "Trace ID" }
        ]
      },
      "post": {
        "tags": ["Users"],
        "summary": "Create user",
        "requestBody": {
          "content": {
            "application/json": {}
          }
        }
      }
    }
  }
}
Limitations: OpenAPI import does not support auth configuration, body content, or collection variables. It only imports paths, methods, parameters (query/header), and body type. For full-featured test files, use Postman v2.1 format.

5. Format 3: cURL Command

For quick single-request imports, a cURL command string can be used. This is the simplest but most limited format.

curl -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOi..." \
  -d '{"name":"Alice","email":"alice@example.com"}' \
  https://api.example.com/users

Supported cURL Options

OptionParsed As
http(s)://...Request URL
-X METHODHTTP method
-H "Key: Value"Header
-d 'data'Raw body (POST if no method specified)
--data-raw 'data'Raw body (same as -d)
--url URLAlternative URL specification
Limitations: cURL import creates a single request only (no folders, no variables, no collection-level auth).

6. Variable System & Request Chaining

Variables are the key to request chaining — using the output of one request as the input of another (e.g., login → extract token → use token in subsequent requests).

6.1 Defining Variables

Define variables at the collection level:

"variable": [
  { "key": "base_url", "value": "https://api.example.com" },
  { "key": "token", "value": "" },
  { "key": "user_id", "value": "" }
]

6.2 Using Variables

Reference variables with double curly braces {{variable_name}} in:

  • URL: "raw": "{{base_url}}/users"
  • Headers: "value": "Bearer {{token}}"
  • Body: "raw": "{\"user_id\":\"{{user_id}}\"}"
  • Auth: "value": "{{token}}"

6.3 Built-in Dynamic Functions

These can be used anywhere variables are supported:

FunctionDescriptionExample Output
{{$timestamp}}Current Unix timestamp1720000000
{{$uuid}}Random UUIDa1b2c3d4-e5f6-...
{{$randomInt}}Random integer42
{{$datetime}}Current ISO datetime2026-07-07T12:00:00Z

6.4 Request Chaining Pattern

While Jcurl does not auto-extract variables from responses via import files, the developer can:

  1. Run the login request
  2. Copy the token from the response
  3. Update the token variable in the environment/variables panel
  4. All subsequent requests using {{token}} will automatically use the new value
Pro tip: Define all chainable variables with empty values in the import file. The developer fills them in after running the first request. This makes the test flow self-documenting.

8. AI Prompt Template

When you want an AI to generate an import file for your project, use this prompt:

Prompt for AI

You are helping me build a project. I need you to generate a Jcurl 
auto-test import file (Postman v2.1 JSON format) for all the API 
endpoints in my project.

Requirements:
1. Output a single valid JSON file in Postman v2.1 collection format
2. Set "info.name" to the project name
3. Define collection-level variables:
   - "base_url" with the server URL
   - "token" with empty value (for auth chaining)
   - Any other IDs needed for request chaining
4. Group endpoints into folders by module/controller
5. For each endpoint include:
   - method, url (use {{base_url}} prefix)
   - headers (Content-Type for body requests)
   - body (raw JSON for POST/PUT/PATCH)
   - auth (bearer with {{token}} for protected routes)
6. Use {{variable}} syntax for chainable values
7. Use {{$uuid}}, {{$timestamp}}, {{$randomInt}} for test data
8. Make sure the JSON is valid and complete

Here are my API endpoints:
[describe or paste your routes/controllers/OpenAPI spec here]

After the AI generates the JSON, save it as .json and import into Jcurl via the Import menu.

9. Generation Checklist

Before finalizing your import file, verify:

Mandatory

#Check
1Valid JSON (no trailing commas, no comments)
2info.name is set
3item array is non-empty
4Each request has method and url
5URL uses {{base_url}} variable prefix

Recommended

#Check
6Endpoints grouped into folders by module
7base_url variable defined at collection level
8token variable defined (empty) for auth chaining
9Protected routes have auth.type: bearer with {{token}}
10POST/PUT/PATCH requests have Content-Type: application/json header
11Body raw is a valid JSON string (escaped quotes)
12Test data uses dynamic functions ({{$uuid}}, {{$randomInt}})

Common Mistakes to Avoid

MistakeFix
Unescaped quotes in body rawUse \" inside JSON strings
Missing mode in bodyAlways set "mode": "raw"
Hardcoded URLsUse {{base_url}} variable
Hardcoded tokensUse {{token}} variable
Folder without item arrayFolders must have item: [...]
Request inside folder missing requestEach request item needs a request object

1. 概述 — 这是什么?

本文档是写给 AI 助手（以及人类开发者）的规范。它教你如何生成 自动测试导入文件 — 一种结构化的 JSON 或文本文件，描述 HTTP API 测试用例，可直接导入 Jcurl 执行。

使用场景: 如果你是一个 AI，正在帮助某人构建项目（REST API、Web 服务、微服务），你可以按照本规范生成一个导入文件。开发者将其导入 Jcurl 后，即可立即拥有所有 API 端点的测试用例 — 无需手动配置。

工作流程

  1. AI 阅读项目的 API 定义（路由、控制器、OpenAPI 文档等）
  2. AI 按照本规范生成 Postman v2.1 格式的 .json 文件
  3. 开发者在 Jcurl 中: 导入 → 选择文件
  4. 所有端点作为集合出现，可直接执行

2. 支持的格式

Jcurl 支持三种导入格式。强烈推荐 Postman v2.1，因为它支持的功能最多。

格式扩展名支持功能推荐度
Postman v2.1.json文件夹、变量、认证、所有 body 类型、查询参数、请求头最佳
OpenAPI 3.0.json按 tag 分组的路径、参数、请求体类型良好
cURL.txt仅单个请求、请求头、body、方法有限

3. 格式一: Postman v2.1 JSON 推荐

这是最完整的格式，支持嵌套文件夹、集合级变量、每请求独立认证和所有 body 类型。

3.1 顶层结构

{
  "info": {
    "name": "我的 API 测试",
    "schema": "https://schema.getpostman.com/json/collection/v2.1.0/collection.json",
    "_postman_id": "任意唯一字符串"
  },
  "variable": [ ... ],
  "item": [ ... ]
}
字段类型必填说明
info.namestring是集合名称（显示在 Jcurl 中）
info.schemastring否Postman schema URL（兼容性用）
info._postman_idstring否唯一 ID（省略则自动生成）
variablearray否集合级变量（见 §6）
itemarray是顶层项目（文件夹或请求）

3.2 项目类型: 文件夹 vs 请求

item 数组中的每个项目要么是 文件夹（包含嵌套项目），要么是 请求（包含 request 对象）。

文件夹

{
  "name": "用户管理",
  "item": [
    { "name": "登录", "request": { ... } },
    { "name": "获取资料", "request": { ... } }
  ]
}

请求

{
  "name": "创建用户",
  "request": {
    "method": "POST",
    "url": { ... },
    "header": [ ... ],
    "body": { ... },
    "auth": { ... }
  }
}
判断规则: 如果项目有 item 数组且没有 request 字段，则是文件夹。如果有 request 字段，则是请求。

3.3 URL 结构

url 字段可以是简单字符串或带查询参数的对象:

简单 URL

"url": "https://api.example.com/users"

带查询参数的 URL

"url": {
  "raw": "https://api.example.com/users?page=1&limit=20",
  "query": [
    { "key": "page", "value": "1" },
    { "key": "limit", "value": "20", "disabled": false }
  ]
}
字段类型说明
rawstring完整 URL（用作请求 URL）
queryarray查询参数列表
query[].keystring参数名
query[].valuestring参数值
query[].disabledbooleantrue 表示禁用该参数（默认: false）

3.4 请求头

"header": [
  { "key": "Content-Type", "value": "application/json" },
  { "key": "Authorization", "value": "Bearer {{token}}", "disabled": false },

]
字段类型说明
keystring请求头名称
valuestring请求头值
disabledbooleantrue 表示禁用该头（默认: false）

  }
}
限制: OpenAPI 导入不支持认证配置、body 内容或集合变量。仅导入路径、方法、参数（query/header）和 body 类型。如需完整测试文件，请使用 Postman v2.1 格式。

5. 格式三: cURL 命令

用于快速导入单个请求，是最简单但最受限的格式。

curl -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOi..." \
  -d '{"name":"Alice","email":"alice@example.com"}' \
  https://api.example.com/users

支持的 cURL 选项

选项解析为
http(s)://...请求 URL
-X METHODHTTP 方法
-H "Key: Value"请求头
-d 'data'原始 body（未指定方法时默认 POST）
--data-raw 'data'原始 body（同 -d）
--url URL替代的 URL 指定方式
限制: cURL 导入仅创建单个请求（无文件夹、无变量、无集合级认证）。

6. 变量系统与请求链

变量是实现 请求链 的关键 — 将一个请求的输出作为另一个请求的输入（例如: 登录 → 提取 token → 在后续请求中使用 token）。

6.1 定义变量

在集合级定义变量:

"variable": [
  { "key": "base_url", "value": "https://api.example.com" },
  { "key": "token", "value": "" },
  { "key": "user_id", "value": "" }
]

6.2 使用变量

用双花括号 {{variable_name}} 在以下位置引用变量:

  • URL: "raw": "{{base_url}}/users"
  • 请求头: "value": "Bearer {{token}}"
  • Body: "raw": "{\"user_id\":\"{{user_id}}\"}"
  • 认证: "value": "{{token}}"

6.3 内置动态函数

可在任何支持变量的地方使用:

函数说明示例输出
{{$timestamp}}当前 Unix 时间戳1720000000
{{$uuid}}随机 UUIDa1b2c3d4-e5f6-...
{{$randomInt}}随机整数42
{{$datetime}}当前 ISO 日期时间2026-07-07T12:00:00Z

6.4 请求链示例

虽然 Jcurl 不会通过导入文件自动从响应中提取变量，但开发者可以:

  1. 运行登录请求
  2. 从响应中复制 token
  3. 在环境/变量面板中更新 token 变量
  4. 所有使用 {{token}} 的后续请求将自动使用新值
建议: 在导入文件中将所有可链接变量定义为空值。开发者在运行第一个请求后填入值。这样测试流程就自带文档说明。

8. AI 提示词模板

当你想让 AI 为你的项目生成导入文件时，使用以下提示词:

给 AI 的提示词

你正在帮助我构建一个项目。我需要你为项目中所有 API 端点
生成一个 Jcurl 自动测试导入文件（Postman v2.1 JSON 格式）。

要求:
1. 输出一个 Postman v2.1 集合格式的有效 JSON 文件
2. 设置 "info.name" 为项目名称
3. 定义集合级变量:
   - "base_url" 服务器 URL
   - "token" 空值（用于认证链接）
   - 其他请求链接所需的 ID
4. 按模块/控制器将端点分组为文件夹
5. 每个端点包含:
   - method, url（使用 {{base_url}} 前缀）
   - headers（body 请求需要 Content-Type）
   - body（POST/PUT/PATCH 用 raw JSON）
   - auth（受保护路由用 bearer + {{token}}）
6. 可链接的值使用 {{variable}} 语法
7. 测试数据使用 {{$uuid}}, {{$timestamp}}, {{$randomInt}}
8. 确保 JSON 有效且完整

以下是我的 API 端点:
[在此描述或粘贴你的路由/控制器/OpenAPI 规范]

AI 生成 JSON 后，保存为 .json 文件，在 Jcurl 中通过导入菜单导入。

9. 生成检查清单

在最终确认导入文件前，请检查:

必须项

#检查项
1JSON 格式有效（无尾逗号、无注释）
2info.name 已设置
3item 数组非空
4每个请求有 method 和 url
5URL 使用 {{base_url}} 变量前缀

建议项

#检查项
6端点按模块分组为文件夹
7集合级定义了 base_url 变量
8定义了 token 变量（空值）用于认证链接
9受保护路由有 auth.type: bearer + {{token}}
10POST/PUT/PATCH 请求有 Content-Type: application/json 头
11body raw 是有效的 JSON 字符串（转义引号）
12测试数据使用动态函数（{{$uuid}}, {{$randomInt}}）

常见错误

错误修复
body raw 中引号未转义在 JSON 字符串中使用 \"
body 缺少 mode始终设置 "mode": "raw"
URL 硬编码使用 {{base_url}} 变量
token 硬编码使用 {{token}} 变量
文件夹缺少 item 数组文件夹必须有 item: [...]
文件夹内请求缺少 request每个请求项需要有 request 对象
