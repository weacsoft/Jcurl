# JPostman AI 工作指南

> 本文档定义 AI 在参与 JPostman 项目开发时必须遵循的工作方法论。
> 方法论源自 [mattpocock/skills](https://github.com/mattpocock/skills) 工程技能集,针对 JPostman 项目做了适配。

---

## 一、核心理念

软件开发中最大的失败模式不是代码写不出来,而是**对齐失败** —— 你以为 AI 理解了你的需求,实际上它理解的是另一个东西。其次才是代码质量、架构腐化、反馈缺失。

JPostman 的 AI 工作方法论围绕四个问题展开:

| 问题 | 应对方法 |
|------|----------|
| AI 没理解我想要什么 | **Grilling**: 开始前逐个问题拷问,走遍决策树每个分支 |
| AI 说话太啰嗦,术语混乱 | **Domain Modeling**: 维护 CONTEXT.md 术语表,用项目语言说话 |
| 代码写出来不能跑 | **TDD**: 红绿重构,一个测试一个实现,小步前进 |
| 代码越写越乱 | **Codebase Design**: 设计深度模块,小接口藏大实现 |

---

## 二、需求对齐:Grilling(拷问式问答)

### 什么时候用

每次开始一个新步骤(对应《JPostman开发规划》中的 13 个步骤)之前,如果需求存在任何模糊点,先进行 Grilling。

### 怎么做

1. **一次只问一个问题**。连续抛出多个问题会让人不知所措。
2. **每个问题附带推荐答案**。用户只需确认或修正,降低沟通成本。
3. **能从代码库找到答案的,直接查代码,不要问用户**。
4. **走遍决策树的每个分支**,逐个解决决策之间的依赖关系。
5. **持续直到达成共识** —— 每个分支的决策都 resolved。

### 示例

> 问:步骤 4 的 HTTP 引擎,SSL 证书验证策略我推荐"默认信任所有证书"(本地调试工具常面对自签名证书)。你同意吗?还是需要提供一个"严格模式"开关?
>
> (等用户回答后,再问下一个问题)

## 三、领域建模:维护 CONTEXT.md

### 文件结构

```
JPostman/
├── CONTEXT.md          # 领域术语表(只有术语定义,不含实现细节)
├── docs/
│   └── adr/            # 架构决策记录
│       ├── 0001-选择sqlite而非h2.md
│       └── 0002-前端用cdn而非构建工具.md
└── ...
```

### CONTEXT.md 规则

- CONTEXT.md 是**术语表**,不是规格说明,不是实现笔记
- 每个术语一行:术语名 + 定义
- 完全不包含实现细节(不写"用 JPA 注解"、"用 OkHttp")
- 当术语在对话中被 resolved 时,**立即更新**,不要攒着批量写

### ADR(架构决策记录)规则

只在**三个条件同时满足**时才创建 ADR:

1. **难以逆转** —— 改主意的成本很高
2. **没有背景会令人困惑** —— 未来的人会问"为什么这么做"
3. **真实权衡的结果** —— 有明确的备选方案,选了一个有具体理由

JPostman 目前已知的 ADR 候选:
- 为什么选 SQLite 而非 H2(单文件、用户熟悉、便于备份)
- 为什么前端用 CDN 而非构建工具(简易定位、Java 技术栈友好)
- 为什么选 OkHttp 而非 Java HttpClient(拦截器、HTTP/2、生态)

### 术语锐化

当用户使用模糊或歧义的词时,AI 应主动提出精确的规范术语:

> 你说的"请求",是指 RequestItem(保存的请求定义)还是 History(一次实际执行的记录)?这两个是不同的东西。

## 四、任务拆分:垂直切片(Tracer Bullet)

### 核心原则

每个任务必须是一个**垂直切片**:贯穿所有集成层端到端(schema → API → UI → 测试),而不是某一层的水平切片。

```
错误(水平切片):
  任务1: 建所有表
  任务2: 写所有 Controller
  任务3: 写所有 Service
  任务4: 写所有前端

正确(垂直切片):
  任务1: 项目管理全链路(表 + API + 界面 + 测试)
  任务2: 请求执行全链路(表 + 引擎 + API + 界面 + 测试)
  任务3: 历史记录全链路(表 + API + 界面 + 测试)
```

### 每个切片必须满足

- 贯穿所有层:数据库 Schema、后端 API、前端界面、测试
- 完成后可独立演示或验证
- 有明确的验收标准(复选框列表)

### 预重构(Prefactoring)

如果某个切片需要先调整现有代码才能更容易实现,先做预重构:"让改变变容易,然后做容易的改变。"

### 与开发规划的关系

《JPostman开发规划》中的 13 个步骤已经按垂直切片思路组织。但每个步骤内部也应保持垂直切片的节奏 —— 不要一口气写完所有实体再写所有 Controller,而是"一个功能模块:实体 → Repository → Service → Controller → 测试"这样推进。

## 五、测试驱动:TDD(红绿重构)

### 核心原则

测试验证**行为**,不验证**实现**。代码内部可以完全重写,测试不应该变。好测试读起来像规格说明:"用户能用 JSON Body 发送 POST 请求"。坏测试在你重构时挂掉,但行为没变。

### 反模式:水平切片

**不要先写完所有测试,再写所有实现。** 这样会产生垃圾测试 —— 你测的是想象中的行为,不是真实行为。

```
错误(水平):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确(垂直):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  ...
```

### 工作流

1. **规划**: 确认接口长什么样、哪些行为最值得测、识别深度模块、为可测试性设计接口
2. **示踪弹**: 写第一个测试(RED) → 写最小实现让它通过(GREEN),证明端到端通路打通
3. **增量循环**: 每个剩余行为,RED → GREEN,一次一个测试
4. **重构**: 全绿后才重构 —— 提取重复、深化模块、运行测试

### 每个循环的检查清单

- [ ] 测试描述行为,不是实现
- [ ] 测试只用公共接口
- [ ] 测试能扛内部重构
- [ ] 代码刚好让这个测试通过
- [ ] 没有投机性的额外功能

### JPostman 的测试策略

- Service 层: JUnit 5 + Spring Boot Test,测试业务逻辑行为
- HTTP 引擎: 用 MockWebServer 模拟目标服务器,不依赖外部 API
- 变量解析: 纯单元测试,验证 `{{key}}` 替换逻辑和多级查找
- Swing 界面: 暂不强制自动化测试,以手动验证为主(简易项目定位);关键交互逻辑可提取到独立类中做单元测试

## 六、代码库设计:深度模块

### 术语(必须一致使用,不要替换)

| 术语 | 定义 |
|------|------|
| **Module(模块)** | 任何有接口和实现的东西。函数、类、包、跨层切片都算 |
| **Interface(接口)** | 调用者必须知道的一切:类型签名、不变量、顺序约束、错误模式、配置要求、性能特征 |
| **Implementation(实现)** | 模块内部的代码体 |
| **Depth(深度)** | 每单位接口学习成本能调用的行为量。小接口 + 大实现 = 深;大接口 + 小实现 = 浅 |
| **Seam(接缝)** | 可以不修改原地代码就改变行为的位置。接口所在的地方 |
| **Adapter(适配器)** | 在接缝处满足接口的具体实现 |
| **Leverage(杠杆)** | 调用者从深度中获得的好处:学一点接口,用很多能力 |
| **Locality(局部性)** | 维护者从深度中获得的好处:改一处,处处生效 |

### 设计原则

1. **深度优先**: 小接口藏大实现。问自己:能不能减少方法数?能不能简化参数?能不能把更多复杂度藏进去?
2. **删除测试**: 想象删掉这个模块。如果复杂度消失了,说明它只是直通(pass-through);如果复杂度在 N 个调用者处重现,说明它在发挥作用
3. **接口就是测试面**: 调用者和测试穿过同一个接缝。如果你需要越过接口去测,说明模块形状不对
4. **一个适配器 = 假想的接缝;两个适配器 = 真实的接缝**。没有真实变化就不要引入接缝

### 为可测试性设计

1. **接受依赖,不要创建依赖** —— 通过参数/构造器注入,不要在方法内 new
2. **返回结果,不要产生副作用** —— 纯函数更容易测
3. **小表面积** —— 方法越少测试越少,参数越少 setup 越简单

### JPostman 中的深度模块示例

`HttpEngineService` 应该是一个深度模块:
- 接口小: `ResponseData execute(RequestConfig config)`
- 实现深: 内部处理变量替换、URL 拼接、Header 设置、Body 构建(form-data/urlencoded/raw/binary)、Auth 注入、超时控制、SSL 信任、异常捕获、前置/后置插件钩子调用、历史记录写入

`VariableResolver` 应该是一个深度模块:
- 接口小: `String resolve(String raw)` 和 `void setScope(Map<String,String> vars)`
- 实现深: 正则匹配 `{{key}}`、多级查找(环境变量→全局变量)、未命中警告收集

## 七、完整工作流程

结合 JPostman 的 13 个步骤,每个步骤的 AI 工作流程:

```
步骤开始
  │
  ▼
1. Grilling(如有模糊点)
  │  逐个问题对齐需求,每个问题带推荐答案
  │  能查代码的不问用户
  │
  ▼
2. 规划切片
  │  确认这个步骤的垂直切片边界
  │  确认测试接缝(seam)在哪里
  │  优先复用现有接缝,接缝越少越好
  │
  ▼
3. 示踪弹(Tracer Bullet)
  │  写第一个测试(RED) → 最小实现(GREEN)
  │  证明端到端通路打通
  │
  ▼
4. 增量 TDD 循环
  │  每个行为: RED → GREEN
  │  一次一个测试,代码刚好通过
  │  不加投机功能
  │
  ▼
5. 重构(全绿后)
  │  提取重复、深化模块、调整接缝
  │  每步重构后跑测试
  │
  ▼
6. 领域建模维护
  │  新术语 resolved → 更新 CONTEXT.md
  │  难逆转+令人困惑+真实权衡 → 创建 ADR
  │  挑战模糊语言,锐化术语
  │
  ▼
7. 验证与交付
  │  对照验收标准逐项确认
  │  与用户确认结果,再进入下一步
```

## 八、红线

1. **不要水平切片**: 不要"先写完所有实体再写所有 Controller"。每个功能模块垂直打通。
2. **不要先写所有测试**: 一个测试一个实现,交替进行。
3. **不要越过接口测试**: 如果需要测私有方法,说明模块设计有问题。
4. **不要在 RED 时重构**: 先到 GREEN,再重构。
5. **不要批量更新 CONTEXT.md**: 术语 resolved 时立即更新,不要攒着。
6. **不要创建不必要的 ADR**: 三个条件(难逆转、令人困惑、真实权衡)缺一不可。
7. **不要用模糊术语**: "请求"是 RequestItem 还是 History?必须明确。
8. **不要一次问多个问题**: Grilling 一次一个问题。
