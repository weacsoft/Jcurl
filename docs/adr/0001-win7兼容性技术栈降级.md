# ADR-0001: Win7 兼容性技术栈降级

## 状态

已接受

## 背景

JPostman 要求最低兼容 Windows 7(服务器最低 Windows Server 2012)。初始技术选型为 SpringBoot 3 + JDK 17 + JavaFX 21 + jaravel-vendor plugin-jar-core,但这与 Win7 兼容性形成三角矛盾:

- SpringBoot 3 强制要求 JDK 17+
- Oracle JDK 17 从未认证过 Windows 7,Server 2012 已标记为不再支持
- JavaFX 版本与 JDK 版本硬绑定,无法跨版本编译

## 决策

将技术栈全面降级,同时采用"高版本开发、低版本运行"策略:

| 层级 | 原方案 | 降级方案 |
|------|--------|---------|
| 开发 JDK | 17 | 11 |
| 编译目标 | 17 | 8 (`--release 8`) |
| 运行 JRE | 17 | 8+ |
| GUI 框架 | JavaFX 21 | Swing + FlatLaf |
| 容器框架 | SpringBoot 3.2 | SpringBoot 2.7 (LTS) |
| 插件系统 | plugin-jar-core 0.1.0 (SB3) | 降级到 SB2 兼容版 |
| 代码编辑器 | RichTextFX | JTextArea |

### 选择 Swing 而非 JavaFX 的理由

JavaFX 版本与 JDK 版本硬绑定:JavaFX 8 只在 JDK 8 中内置,JavaFX 11+ 的 class 文件版本是 Java 11+,无法在 JRE 8 上运行。这意味着无法同时满足"开发用高版本 JDK"和"运行兼容低版本 JRE"。

Swing 内置于所有 JDK 版本(1.2+),不存在版本绑定问题。可以用 JDK 11 开发、`--release 8` 编译、JRE 8 运行,全程无缝。FlatLaf 提供现代扁平外观,弥补 Swing 默认外观老旧的问题。

### 选择 SpringBoot 2.7 而非 3.x 的理由

jaravel-vendor 的 plugin-jar-core 和 plugin-java-core 核心技术(URLClassLoader、ClassLoader 隔离、javax.tools.JavaCompiler)在 JDK 8+ 完全可用,不依赖 JDK 17 新特性。降级到 SpringBoot 2.7 可保持插件系统核心功能,主要改动是 `javax.*` 命名空间(SB3 用 `jakarta.*`),但插件模块不涉及 Web/Servlet,影响极小。

### 选择 JDK 11 开发 + `--release 8` 编译的理由

JDK 11 提供 `var` 局部变量类型推断、HTTP Client API、改进的 Lambda 等现代语法,提升开发体验。`--release 8` 标志确保编译产物兼容 JRE 8,同时 IDE 会将 JDK 11 独有 API 标记为编译错误,防止误用。JRE 8 是 Windows 7 上官方支持最可靠的 Java 运行环境。

## 后果

**正面**:
- Windows 7 / Server 2012 完全兼容
- 开发体验良好(JDK 11 现代语法)
- 一套代码,无需维护双版本
- 插件系统核心功能保留

**负面**:
- JDK 8 语法限制(无 record、sealed class、switch expression、pattern matching)
- Swing 布局不如 JavaFX FXML 声明式直观,需用代码构建
- 需要为 jaravel-vendor 维护 SpringBoot 2 兼容的插件版本
- `javax.*` 与 `jakarta.*` 分裂:jaravel-vendor 主线用 SB3/jakarta,JPostman 用 SB2/javax,插件代码需注意命名空间
